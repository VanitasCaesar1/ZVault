ZVault is configured via environment variables. No config files needed.

## Server Configuration

| Variable | Default | Description |
|----------|---------|-------------|
| `ZVAULT_LISTEN_ADDR` | `127.0.0.1:8200` | Address and port to listen on |
| `ZVAULT_STORAGE_PATH` | `./data` | Path to persistent storage directory |
| `ZVAULT_STORAGE_BACKEND` | `rocksdb` | Storage backend (`rocksdb` or `redb`) |
| `RUST_LOG` | `info` | Log level (`debug`, `info`, `warn`, `error`) |

## Security Defaults

ZVault ships with secure defaults:

- Binds to `127.0.0.1` (localhost only) — set `0.0.0.0` explicitly for network access
- Core dumps disabled (`RLIMIT_CORE=0`)
- Key material pinned in memory with `mlock`
- TLS 1.3 via rustls (no OpenSSL dependency)
- HTTP security headers: `X-Content-Type-Options: nosniff`, `X-Frame-Options: DENY`, `Cache-Control: no-store`

## Storage Backends

### RocksDB (default)

High-performance embedded key-value store. Best for production.

```bash
ZVAULT_STORAGE_BACKEND=rocksdb
ZVAULT_STORAGE_PATH=/data/vault
```

### Redb

Pure-Rust embedded database. No C dependencies. Good for development or minimal deployments.

```bash
ZVAULT_STORAGE_BACKEND=redb
ZVAULT_STORAGE_PATH=/data/vault
```

## Production Checklist

- [ ] Set `ZVAULT_LISTEN_ADDR=0.0.0.0:8200` (or use a reverse proxy)
- [ ] Use a persistent volume for `ZVAULT_STORAGE_PATH`
- [ ] Put a TLS-terminating reverse proxy in front (nginx, Caddy, Cloudflare)
- [ ] Store unseal keys in separate secure locations
- [ ] Create scoped tokens — don't use the root token for daily operations
- [ ] Set up monitoring on the `/v1/sys/health` endpoint
- [ ] Back up the storage directory regularly
